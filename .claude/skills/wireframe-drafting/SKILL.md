# 와이어프레임 초안

**목적**: 디자인 확정 전, 유저 플로우를 *동작하는 저충실도 화면*으로 만들어 배포한다. 기획/디자이너가 실제로 눌러보며 플로우에 공감·피드백하게 하는 게 목표. **시각 완성도가 아니라 구조·흐름·인터랙션**이 산출물.

## 핵심 원칙
- **디자인 가이드 없이 진행** — Figma 토큰·디자인 시스템 의존 금지(아직 없음). 중립적 기본 스타일만.
- **더미/목 데이터** — API 미연동. 로컬 목 데이터로 화면과 전이를 보여준다.
- **버릴 코드 전제** — 초안은 교체될 수 있음. 과한 추상화·최적화·기능 추가 금지.
- **플로우 우선** — 화면 간 이동, 빈/로딩/에러 상태, CRUD 진입점이 *눌리는지*가 중요.

## 스타일 규약 (저충실도)
- Tailwind 기본 유틸리티로 레이아웃만 (간격·정렬·테두리). 색·타이포는 시스템 기본/회색조.
- **토큰 화이트리스트 규칙 면제(초안 한정)** — raw 값·arbitrary value 허용. 단 디자인 확정 후엔 figma-implementer가 토큰 기반으로 다시 만든다.
- placeholder 박스·라벨로 미정 영역 표시 (예: `[이미지]`, `[차트]`).

## 코드 규약 (유지)
- `any` 금지 / barrel export 금지 / hooks 순서 / 모바일 퍼스트 레이아웃은 그대로.
- 더미 데이터는 한 곳(예: `_mocks/`)에 모아 실제 API 교체 지점을 명확히.

## 위치·표시
- 초안 산출물은 **명시적으로 wireframe임을 표시**한다(파일/라우트 네이밍).
- `TODO(✍️):` 초안 라우트 구조(예: `(wireframe)` 그룹) — 도메인/렌더링 전략 확정 시 픽스.

## 배포
- 배포는 외부로 나가는 작업 → 사용자 확인 게이트. diff-organizer/배포 절차를 따른다.

## 전이 (초안 → 정식)
- 디자인 확정 시: figma-implementer가 토큰 기반 재구현, api-developer가 더미→실 API 교체, design-reviewer 검수.
